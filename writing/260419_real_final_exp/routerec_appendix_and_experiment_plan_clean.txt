RouteRec appendix/실험 계획 정리본
====================================

목적
----
이 문서는 현재 본문이 어느 정도 정리된 상태에서,
1. appendix를 어떤 순서와 역할로 구성할지,
2. 각 appendix 섹션에 figure/table을 어떻게 배치할지,
3. 추가로 어떤 실험을 넣어야 본문 Q1~Q5를 가장 잘 보강할지,
4. 실제 실험을 서버 2대(8 GPU + 8 GPU)에서 어떻게 나눠 돌리면 효율적인지,
5. 논문에 실험 환경을 어떻게 적으면 좋을지

를 한 번에 정리한 문서다.

핵심 원칙은 다음과 같다.

- appendix는 “남는 실험 모음”이 아니라 본문 Q1~Q5를 받치는 증거 패키지여야 한다.
- 본문과 연결되는 섹션부터 먼저 구현하고, 시간이 남으면 후순위 섹션을 채운다.
- 각 appendix 섹션은 가능하면 table 1개 + figure 1개(또는 figure 1개만) 수준으로 정리해 과밀도를 피한다.
- 구조/설계, sparse routing, cost, semantics, qualitative evidence를 분리해서 읽기 쉽게 만든다.


0. 본문과 appendix의 연결 구조
-----------------------------

본문 질문 구조:
- Q1. RouteRec가 성능이 되는가?
- Q2. 무엇이 routing을 control해야 하는가?
- Q3. 왜 최종 RouteRec 설계가 이 구조여야 하는가?
- Q4. sparse RouteRec는 practical한가?
- Q5. router는 실제로 무엇을 배우는가?

appendix는 이 질문들을 순서대로 받쳐야 한다.

권장 대응:
- A: Method exact instantiation
- B: Datasets / protocol / full results
- C: special bins (어디서 gain이 나는가)
- D: framework / structural ablations
- E: sparse routing / top-k variants
- F: objective variants
- G: full cost summary
- H: final routing diagnostics
- I: extended behavioral slices
- J: targeted interventions + qualitative cases
- K: optional transfer

즉 appendix도 본문 논리와 비슷하게
설정 -> 결과 확장 -> 구조 정당화 -> sparse design -> cost -> semantics
순으로 읽히게 만드는 것이 좋다.

----------------------

- 실험 구현 우선순위도 명확해진다.


2. Appendix A: Exact RouteRec Instantiation
------------------------------------------

역할:
- 본문 method를 아주 길게 반복하지 않으면서,
  final selected paper setting이 무엇인지 빠르게 확인하게 해주는 섹션.

중요:
- A.2를 길게 쓸 필요는 없다.
- 코드 공개 예정이므로, appendix는 “실행 규칙 최소 설명 + 테이블” 정도면 충분하다.

권장 배치:
- 짧은 opening paragraph
- Table A1: Fixed main configuration

- Table A3: Representative cue families
기존 table들 잘 유지하되 layout만 좀 더 개선. 가시성 좋도록

1. macro/mid는 sequence-wise broadcast, micro는 position-wise
2. top-k mask 후 renormalization
3. grouping field unavailable 시 focus cue fallback / zero-fill 처리

즉 A는 “코드 없이도 paper setting의 final instantiated shape는 이해되게”만 만들면 충분하다.

6. Appendix E: Sparse Routing Design Variants
--------------------------------------------

역할:
- 본문 Q3(b)와 Q4를 동시에 받침
- top-k sparse design이 arbitrary choice가 아니라는 걸 보여주는 핵심 appendix

중요:
- 여기서는 variant 이름을 더 친절하게 써야 한다.
- 예: “2/2”보다 “Top-2 groups, Top-2 experts per group”처럼 써야 독자가 바로 이해한다.

권장 variant set:
1. Dense full mixture
   - group / expert 모두 dense
2. Flat sparse top-6
   - hierarchy 없이 전체 expert 중 top-6
3. Top-4 groups, Top-2 experts per group
   - total 8 active experts
   - group commitment 약함, compute budget 큼
4. Top-2 groups, Top-4 experts per group
   - group commitment 강하지만 group 안 capacity 큼
5. Top-3 groups, Top-2 experts per group   ← main setting
   - total 6 active experts
6. Top-2 groups, Top-1 expert per group
   - 매우 sparse, 과도하게 restrictive할 가능성
7. Optional: Top-3 groups, Top-3 experts per group
   - higher active compute budget reference

권장 이유:
- 2/2, 3/2, 4/2, 3/3 같은 표기보다
  “group selectivity vs intra-group capacity” tradeoff가 더 직관적으로 보인다.
- 4/2, 2/4, 3/2, 2/1은 의도가 눈에 보인다.
  - 4/2: group commitment 약화
  - 2/4: 적은 group + 큰 intra-group capacity
  - 3/2: balanced main setting
  - 2/1: 과도하게 sparse

권장 배치:
- Table E1: sparse routing variant summary
  - Variant name
  - Active experts
  - MRR@20
  - Infer-time ratio
  - Observation
- Figure E1: 2-panel diagnostics across sparse variants
  - (a) route entropy by stage
  - (b) effective expert count by stage
- optional Figure E2:
  - selected variants(dense / main sparse / very sparse)의 stage-wise usage heatmap

여기서 꼭 넣으라고 할 diag 로그:
- route entropy
- effective expert count
- top-1 expert share / route concentration
- dead expert count / underused expert count
- group usage balance
- stage별 route sharpness
- infer-time ratio

설명 포인트:
- 이 섹션은 단순 top-k sweep이 아니라
  “quality–specialization–efficiency tradeoff”를 보여주는 섹션이다.
- 본문 Q3에서 hierarchical sparse routing이 중요하다고 말한 걸 더 자세히 수치화하는 역할도 한다.
- 본문 Q4 cost claim도 이 섹션과 연결된다.

14. 서버 2대(8 GPU + 8 GPU)로 실험 나누는 권장 방식
---------------------------------------------------

목표:
- 서버 2대에서 appendix 실험을 반반 나눠 효율적으로 돌리기
- 각 process는 GPU 1개만 사용
- 논문에는 “RTX 3090에서 실험”이라고만 쓰고,
  여러 GPU는 여러 실험 병렬 실행용이었다고 이해하면 됨

중요:
- 한 실험 안에서 data parallel / model parallel을 한 것은 아님
- “x8 GPU 학습”이 아니라,
  여러 독립 run을 GPU별로 분산 실행한 것 (기존 코드들 보면 이해 가능)

권장 분할 방식:
----------------
서버 A (GPU 0-7): 구조/설계/메인 supporting ablations
- D: framework / structural ablations
- E: sparse routing variants
- J 일부: targeted interventions
- G 일부: RouteRec/FAME/SASRec cost benchmark

서버 B (GPU 0-7): analysis-heavy / supplementary / aggregate supports
- C: special bins
- H: routing diagnostics
- I: extended slice results
- F: objective variants
- G full baseline cost benchmark
- J qualitative export / case extraction

왜 이렇게 나누나:
- 서버 A는 “추가 train이 필요한 실험” 위주
- 서버 B는 “기존 checkpoint를 많이 읽고 aggregate/export/benchmark하는 작업” 위주

더 구체적 권장:
---------------
1. 구조 sweep / sparse routing variant는 서버 A
   - train-heavy
   - run 수가 많고 각 run이 오래 걸림

2. diagnostics / bins / slice / intervention export는 서버 B
   - 학습보다는 evaluation/export 성격이 강함
   - checkpoint 재사용이 많음

3. cost benchmark는 혼합 가능
   - training 필요 없고 checkpoint benchmark라 가볍지만,
     baseline 수가 많아 모으면 시간 걸릴 수 있으므로 B 쪽이 더 적합

코드 설계 지시:
--------------
- appendix 질문별로 launcher를 분리하되,
  “candidate rows -> manifest -> run_jobs” 형태를 유지
- 각 launcher는 서버마다 subset만 실행 가능하게 filter 인자 제공
  - 예: --only-panels D,E,J
  - 예: --only-variants topk,layout
  - 예: --datasets Beauty,KuaiRec,...
- run id / variant label / appendix section tag를 CSV에 같이 저장
- plotting notebook는 section별 summary CSV를 바로 읽게 구성

예시 분할 태그:
- section_tag = D_structural
- section_tag = E_sparse
- section_tag = F_objective
- section_tag = C_special_bins
- section_tag = H_diag
- section_tag = I_slices
- section_tag = J_intervention
- section_tag = G_cost

이렇게 해두면 서버 A/B가 각각 다른 section_tag만 돌리고,
나중에 notebook가 자동으로 section별 summary를 읽게 만들 수 있다.

실행 운영 팁:
------------
- 서버 A:
  - GPU 0-7에 D/E train job을 분산
  - 오래 걸리는 run부터 먼저 배치
- 서버 B:
  - 남는 GPU에 F objective variants 일부 같이 배치
  - 나머지는 export/benchmark scripts 병렬 실행
- 먼저 A에서 main-setting checkpoint를 충분히 확보한 뒤
  B가 그 checkpoint들로 diagnostics/interventions를 생성하게 하면 병목이 줄어든다.

17. 최종 한 줄 정리
-------------------
