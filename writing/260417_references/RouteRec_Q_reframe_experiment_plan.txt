RouteRec experiment question 재설계 및 main/appendix 보완 계획

[총평]
지금 본문 5.2~5.5의 Q 흐름은 이미 꽤 괜찮다.
현재 실험 본문은 대략 아래 질문들에 대응한다.
- Q1. Does behavior-guided routing help overall?
- Q2. What should control routing?
- Q3. Why is the staged structure effective?
- Q4. Are lightweight cues enough?

이 4개 자체는 나쁘지 않다. 다만 reviewer 관점에서 보면, 지금 가장 강하게 남는 의문은 아래 둘이다.
1) "정말 이 routing이 네가 말하는 behavioral regimes를 반영하냐?"
2) "method의 세 축(feature, stage, routing composition) 중 뭐가 핵심이고, 셋 다 필요한가?"

즉 지금의 Q1~Q4는 accuracy / design / portability는 보지만,
- semantic validity
- method three-part justification
이 조금 약하다.

따라서 가장 추천하는 방향은
- Q1~Q4를 완전히 버리기보다,
- Q3를 method의 3축 정당화로 더 강하게 바꾸고,
- Q5를 새로 추가해서 routing semantics를 본문으로 올리는 것
이다.

==================================================
1. reviewer가 method를 읽으면서 실제로 드는 의문
==================================================

method를 읽는 reviewer의 머릿속 질문은 대체로 아래 순서로 생긴다.

(1) "그래서 성능은 진짜 오르나?"
-> Q1 overall performance 필요.

(2) "hidden-state router보다 behavior cue router가 왜 더 낫다는 거지?"
-> Q2 control-source comparison 필요.

(3) "macro/mid/micro를 왜 셋으로 나눴지? 아무 expert insertion이면 안 되나?"
-> stage/layout ablation 필요.

(4) "Tempo / Focus / Memory / Exposure 같은 cue family가 진짜 의미 있나?"
-> cue-family ablation 필요.

(5) "근데 그 routing이 정말 semantic한지 어떻게 아나? 그냥 성능이 좋아졌을 뿐 아니냐?"
-> 이 질문이 지금 가장 중요함.
-> feature bucket / slice / case study / route concentration 같은 분석이 본문에 하나는 있어야 함.

(6) "이 논문의 핵심은 feature construction, stage decomposition, routing composition인데, 셋 다 필요한가?"
-> method의 3파트를 직접 대응시키는 structural figure/table이 있으면 매우 좋음.

(7) "objective(CE + consistency + z-loss)는 왜 이런 조합이지?"
-> 이건 appendix로 충분하지만, 최소한 objective variant table은 있어야 함.

(8) "이 결과가 특정 protocol/sessionization에만 묶인 건 아닌가?"
-> appendix에서 protocol sensitivity 또는 seen-target/all-target 보조 결과가 있으면 좋음.

즉 abstract/introduction/method/experiment의 주장축이 가장 잘 맞는다.

Q2. What should control routing?
Q3. Why are the three design choices of RouteRec necessary?

3. Q3를 어떻게 바꿔야 하나: method의 3축과 직접 연결
==================================================

현재 Figure 4는 mainly stage/layout 쪽이다.
그런데 미팅에서 말한 "method의 3가지 구조 정당화"를 진짜 살리려면,
Q3 자체를 아래처럼 재정의하는 게 좋다.

Q3. Why is RouteRec’s three-part design effective?
  Q3-a. Feature construction: why should routing use compact behavioral cues?
  Q3-b. Stage decomposition: why macro/mid/micro instead of one shared routing scope?
  Q3-c. Routing composition: why the chosen scalar-group conditional composition?

이렇게 되면 method 섹션의 3개 큰 파트와 experiment가 1:1로 맞물린다.
reviewer 입장에서도 "아 이 저자들은 method의 각 설계 choice를 실험으로 확인하려는구나"가 보인다.

[이때 중요한 점]
Q2와 Q3-a가 겹치지 않게 해야 한다.
- Q2는 router input source 비교: hidden-only vs behavior-only vs mixed.
- Q3-a는 behavior feature 설계 자체 비교: full cues vs seq-only vs shuffled/corrupted cues vs family removal.

즉
- Q2 = 무엇으로 routing할 것인가?
- Q3-a = behavior cue를 어떻게 구성해야 하는가?
로 역할을 분리하면 된다.

4.4 Q4. are lightweight cues enough?
--------------------------------------------------

현재 Figure 5 방향은 좋다.
이건 유지 가능.

6. appendix에서 꼭 있으면 좋은 것
==================================================

appendix는 "본문 claim을 더 단단하게 만드는 증거 저장소"로 쓰는 게 좋다.
우선순위별로 정리하면 아래와 같다.

[우선순위 A: 거의 필수]

A1. Objective and regularization variants
- CE only
- +consistency
- +z-loss
- +both (final)
- optional: load balancing or entropy regularizer baseline
보여줄 metric:
- MRR@20 or Avg Rank
- route consistency
- entropy / effective expert count
목적:
- 왜 이 objective가 필요한지 설명

A2. Extended stage/layout ablations
- 지금 appendix B 유지
- 단, route composition variant가 있으면 더 좋음

A3. Routing diagnostics
- expert usage heatmap by stage
- entropy and effective expert count
- stage-wise route consistency
- feature-bucket route pattern
지금 appendix D 방향이 매우 좋음

A4. Behavior-slice analysis (full version)
- dataset별 full slice plot
- 본문엔 compact 버전, appendix엔 full version

[우선순위 B: 있으면 강함]

B1. Protocol sensitivity / evaluation view sensitivity
- seen-target vs all-target or supplementary unseen-target view
- optional: one dataset에서 threshold sensitivity small check
목적:
- 네 protocol이 완전히 brittle하지 않음을 보여줌
주의:
- 이것 때문에 main story가 protocol defense로 흐르지 않게 appendix에 두는 게 좋음

B2. Router composition variants
- flat router
- grouped router
- final scalar-group conditional
- maybe stage-specific bespoke router
이건 구조 정당화에 꽤 좋음

B3. Qualitative case studies
- 2~4 representative sessions
- feature summaries + stage-wise routing + prediction outcome
좋은 경우:
- repeat-heavy
- exploration-heavy
- high-tempo
- stable preference

[우선순위 C: optional]

C1. Seed stability / significance
- dataset별 mean±std or win count over seeds
장점: reviewer 방어에 도움
단점: 실험량 증가

C2. Efficiency / complexity summary
- params
- training time relative to SASRec
- inference overhead rough number
네 paper가 "lightweight"를 주장하니 있으면 좋지만, 지금은 최우선은 아님

C3. Transfer results
- 이미 appendix F로 계획되어 있음
- 있으면 nice bonus, 없어도 paper core는 유지됨
