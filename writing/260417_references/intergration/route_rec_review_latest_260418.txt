1. 총평
============================================================

한 줄 총평:
- front-half와 methodology는 이전보다 확실히 좋아졌고, "논문이 무엇을 하려는가"는 이제 꽤 잘 보인다.
- 하지만 experiment/appendix를 통합하면서 "최종 논문 문장"이 아니라 "설계 메모/방어 논리/작성 계획"이 본문에 꽤 많이 남았다.
- 그래서 지금 버전은 '아이디어 설명'은 좋아졌는데, '완성된 논문'으로 읽을 때는 아직 군데군데 멈칫하게 만드는 부분이 있다.

이전 버전 대비 좋아진 점:
1) challenge 1/2/3 → method 3축 정렬은 유지되고 있다.
2) 4.1 experimental setup은 이전보다 훨씬 방어적이고 투명하다.
3) protocol justification과 tuning fairness를 본문에서 직접 다루려는 방향은 좋다.
4) appendix가 단순 부록이 아니라 "방어 문서" 역할을 하려는 의도가 분명해졌다.

하지만 통합 후 새로 생긴 문제:
1) experiment 파트가 결과 보고보다 "이 섹션이 무엇을 하려는지 설명하는 메타 서술"이 많아졌다.
2) main text figure 설명과 실제 figure/caption이 안 맞는 부분이 생겼다.
3) appendix에는 아직 placeholder / plan 문장 / TODO 느낌 문장이 많이 남아 있다.
4) main text와 appendix 사이의 역할 분담이 오히려 덜 선명해진 부분이 있다.

즉,
- 아이디어 논문으로는 좋아졌고
- reviewer rebuttal 대비용 방어 논리도 들어갔지만
- 최종 제출본으로는 아직 "말할 내용"과 "논문에 실제로 남겨야 할 문장"이 분리되지 않은 상태다.

즉 experiment 파트는
"좋은 기획안"처럼 읽히는 부분이 아직 남아 있다.

3. 가장 큰 문제들
============================================================

3.1 가장 큰 문제: experiment 본문이 너무 메타적이다
현재 4.1~4.6은 결과 보고 + 해석 + reviewer 대응 논리 + 글쓰기 의도가 섞여 있다.

대표적인 패턴:
- "The purpose of Q1 is deliberately narrow."
- "The intent of Q2 is to isolate..."
- "This analysis is intended to support..."
- "The main text is therefore written as an argument rather than as a list of plots..."
- "Once the final numbers are inserted..."
- "the important reading is not whether..."
이런 문장들.

이런 문장은 내부 메모/작문 가이드로는 훌륭하다.
하지만 논문 본문에는 많이 남기면 안 된다.

왜 문제냐:
- 리뷰어는 "결과를 해석하는 논문"을 읽고 싶지, "우리가 이 실험을 왜 배치했는지 설명하는 저자의 메타 코멘터리"를 길게 읽고 싶진 않다.
- 한두 문장 정도는 괜찮지만, 지금은 분량이 꽤 크다.
- 특히 4.2, 4.3, 4.4, 4.5, 4.6 모두 같은 스타일이라 반복감도 생긴다.

수정 방향:
- 각 Q subsection마다 "의도 설명"은 첫 1~2문장으로 줄인다.
- 그 뒤는 바로 결과와 해석으로 들어간다.
- 현재 문장 중
  - purpose / intent / meant to / intended message / once final numbers are inserted
  류 문장은 대부분 삭제 대상이다.
- 본문은 "what we observe" 중심으로,
  appendix는 "why this is a fair and diagnostic setup" 중심으로 나누는 게 좋다.

3.2 figure 설명과 실제 figure/caption이 맞지 않는 부분이 있다
이건 꽤 심각한 통합 버그다.

현재 4.4 Q3의 설명은:
- panel (a): semantic organization of cue space
- panel (b): temporal roles of macro/mid/micro
- panel (c): final router composition

처럼 써져 있다.

그런데 main text의 Figure 4 caption은 여전히
- (a) Stage removal
- (b) Dense replacement versus staged routing
- (c) Router-composition and order variants
이다.

즉, 4.4의 본문 설명은 현재 Figure 4가 아니라, 오히려 Appendix C의 확장 구조 ablation(Figure 7)에 더 가깝다.

이건 리뷰어가 바로 잡아낼 수 있다.
"본문 텍스트와 그림이 안 맞는데?" 라는 반응이 나온다.

수정 방향:
선택지는 둘 중 하나다.

A안)
- Figure 4를 정말 Q3 설명에 맞게 바꾼다.
- 즉 main text Figure 4를
  1) cue semantics
  2) stage semantics
  3) router composition
  이 3축으로 재구성한다.

B안)
- Figure 4는 그대로 두고,
- 4.4 본문을 Figure 4 caption과 정확히 맞춘다.
  즉:
  - (a) stage removal
  - (b) dense replacement vs staged routing
  - (c) router/order variants
  로 다시 써야 한다.

개인적으로는 B안이 더 현실적이다.
이미 지금 구조상 Q3는 "three-part design"을 묻는 것이므로,
- stage removal = stage meaning
- dense replacement vs staged routing = 그냥 expert insertion이 아니라 staged design이라는 점
- router/order variants = arbitrary implementation이 아니라는 점
으로 정리하면 충분히 논리적이다.

3.3 main table / Q1 설명 / appendix table 사이의 일관성이 아직 덜 잠겼다
사용자께서 이미 보신 것처럼, main table은 아직 Improv. 컬럼이다.
그런데 4.2 본문은 "Table 1 ... together with an Avg. Rank summary across all reported rows"라고 쓴다.
이건 현재 표와 맞지 않는다.

또 appendix의 full results table(Table 8)는 caption에서 Avg. Rank를 말하지만,
실제 현재 parsing만 보면 Avg. Rank row가 바로 보이지 않는다.
(실제 PDF 렌더링에는 있을 수도 있지만, 최소한 텍스트/독자 경험상 혼란을 줄 수 있다.)

왜 문제냐:
- reviewer가 표와 본문을 같이 보다가 "지금 본문이 말하는 게 표에 없는데?"라고 느낀다.
- 이런 불일치는 사소한 오타보다 더 신뢰를 깎는다. 논문이 아직 stabilize 안 됐다는 신호이기 때문이다.

수정 방향:
- Table 1을 끝까지 Improv.로 갈 거면, 4.2의 Avg. Rank 언급 삭제.
- Table 1을 Avg. Rank 중심으로 바꿀 거면, באמת table을 바꿔라.
- 둘 다 하고 싶으면
  - main text: Avg. Rank row/summary
  - appendix: per-row Improv. or full table
  로 역할을 나눠라.
- 현재 미팅 기준으로는 main table에 Avg. Rank가 더 잘 맞는다.
  이유:
  1) 이 논문은 모든 dataset에서 absolute SOTA를 주장하기 어렵다.
  2) protocol-sensitive setting이라 평균적 standing을 보여주는 게 더 정직하다.
  3) "improve"는 RouteRec 중심 framing이라 reviewer에게 조금 방어적으로 보일 수 있다.

추천:
- main Table 1: 대표 metric + Avg. Rank 요약
- appendix Table 8: full cutoff grid + maybe bold/underline
- improvement percentage는 본문 해석이나 supplementary sentence로만 남기기

4. method와 experiment 연결에서 더 보완할 점
============================================================

4.1 현재 연결은 좋아졌지만, 약간 과설명이다
좋아진 점:
- 3.2 마지막에 cue contribution을 4.5에서 테스트한다고 연결
- 3.3 마지막에 stage placement/order를 4.4와 Appendix C로 연결
- 3.4 마지막에 alternative routing controls를 4.3으로 연결
이건 매우 좋다.

문제:
- experiment 쪽에서도 같은 연결을 또 길게 설명한다.
- 즉 method에서 이미 "이건 여기서 검증한다"고 했는데,
  experiment가 다시 "Q2는 이 skeptic을 막기 위해..." 식으로 장황해진다.

수정 방향:
- method 끝 문장들은 유지 가능.
- experiment는 그걸 받아서 바로 결과와 해석으로 들어가야 한다.
- method에서 1번, experiment에서 1번만 연결되면 충분하다.
- 지금은 2번 이상 반복되는 느낌이다.

4.2 Q3와 Q4의 경계는 좋아졌지만, 더 날카롭게 할 수 있다
현재 설명상:
- Q3 = design이 meaningful한가
- Q4 = cue availability가 제한돼도 usable한가

이 구분은 좋다.
하지만 본문에서는 아직 조금 겹친다.

리뷰어가 느낄 수 있는 의문:
- Q3 panel (a)가 cue semantics를 건드리면, 그건 사실 Q4와도 가까운 것 아닌가?
- Q4가 availability question이라면, 정말 "semantic family removal"과는 별개로 purely deployability-oriented reduction만 보여주나?

수정 방향:
- Q3는 "semantic/architectural necessity"만.
- Q4는 "availability/portability under missing signals"만.
- 그래서 Q4의 variant naming도
  - no-category
  - no-time
  - sequence-only
  같은 availability-oriented reduction을 전면에 두고,
  Q3에서는
  - shuffled family
  - flat cue bank
  - random grouping
  같은 semantic-structure ablation을 다루는 게 더 깔끔하다.

즉, Q3/Q4의 variant set 자체를 글에서도 더 분명히 분리해줘야 한다.

4.3 Q5는 좋은데, slice definition을 본문에서 너무 늦게 방어한다
Q5는 지금 아주 좋은 방향이다.
이 논문의 motivation(Figure 1)과 가장 잘 연결되는 experiment다.

다만 reviewer는 즉시 이렇게 묻는다.
- repeat-heavy / fast-tempo / focused / exploration-heavy를 어떻게 정의했지?
- 이거 RouteRec feature를 그대로 이름 바꿔서 slice 만든 거 아냐?

현재 4.6에서는 appendix H의 post hoc simple statistics로 built 했다고 말한다.
이건 좋다.
하지만 지금 중요한 만큼, 본문에도 한 문장만 더 구체적으로 써주는 게 좋다.

예:
- repeat-heavy = prefix repeat ratio 상위 quantile
- fast-tempo = mean inter-event gap 하위 quantile
- focused = entropy 하위 quantile
- exploration-heavy = novelty/switching 상위 quantile
같은 식으로, "simple percentile-based statistics" 정도라도 알려주면 좋다.

즉,
- 자세한 정의는 appendix
- 본문에는 "simple externally defined summary statistics" 정도를 한 문장 더
넣으면 reviewer 불안을 더 줄일 수 있다.

5.1 theme/category proxy 부분
여전히 약간 흐리다.

그런데 dataset에 따라 category, theme, venue category, genre 등 종류가 다르다.

예:

5.2 Table 3은 재현성 표인데 아직 final spec이 아니다
지금 Table 3은 reproducibility를 위한 것처럼 보이지만,
실제로는 representative range가 많다.

예:
- Experts per group: C in {2,3,4}
- Router hidden width: range 48–128

이건 search-space summary이지 final spec이 아니다.
Table 2는 fixed main configuration이라고 했는데, Table 3은 다시 range를 준다.
리뷰어는 혼란스럽다.

수정 방향:
두 가지 중 하나로 정리해야 한다.

A안) 정말 reproducibility 표로 갈 거면
- final selected values를 적어라.
- 최소한 main paper runs에 사용한 exact value를 적어라.

B안) search-space summary로 갈 거면
- 표 제목을 바꿔라.
  "Candidate search ranges for implementation-level controls"
  같은 식으로.
- 그리고 별도의 final selected config table을 또 두라.

추천:
- Table 2 = fixed conceptual/main architecture
- Table 3 = exact selected implementation values
- Table 6/7 같은 placeholder는 없애고, 여기로 흡수
이게 제일 깔끔하다.

5.3 dense soft mixing 정당화는 조금 더 짧고 강하게

이 정도면 충분하다.

그리고 experiments나 appendix에서 sparse top-k 이야기를 너무 더 꺼내지 않는 게 좋다.

============================================================
6. experimental setup / protocol justification 쪽 평가
============================================================

이 부분은 이전보다 확실히 좋아졌다.
특히:
- sessionized protocol이 canonical raw-sequence leaderboard가 아니라 assembled protocol이라는 점
- seen-target primary view를 쓰는 이유
- session split / cleanup / temporal order 정당화
- tuning을 protocol 일부로 취급한다는 점
이건 전보다 훨씬 방어력이 있다.

하지만 지금은 조금 과하다.
즉,
좋은 방어 논리를 다 넣으려다가 본문 밀도가 올라간 상태다.

리뷰어 반응 가능성:
- "이 프로토콜이 왜 필요한지는 알겠는데, 설명이 너무 길다."
- "이 논문이 recommender method paper인지, evaluation protocol note인지 순간 헷갈린다."

수정 방향:
4.1은 크게 세 덩어리만 남겨라.

(1) datasets + preprocessing + sessionization
(2) split / seen-target evaluation / transparency on subsets
(3) tuning / selection rule / fairness

그리고 각 덩어리마다
- 핵심 주장 1~2문장
- 세부 근거 1문장
정도로 자르는 게 좋다.

지금처럼
- benchmark philosophy
- tuning philosophy
- writing philosophy
가 한 section 안에 다 들어오면 좀 무겁다.

============================================================
7. appendix 구성 평가
============================================================

현재 appendix 구성 의도는 좋다.
A~I를 보면:
- A: method reproducibility
- B: dataset/protocol/full results
- C: structural ablation
- D: protocol sensitivity
- E: objective variants
- F: efficiency
- G: diagnostics
- H: behavior-slice
- I: transfer

이 자체는 매우 논리적이다.

문제는 아직 "실제 evidence appendix"가 아니라 "planned appendix structure"인 파트가 꽤 있다는 점이다.

좋은 역할 분담:
- A = method spec
- B = protocol defense
- C/E/G/H = claim별 방어
- F = lightweight claim anchor
- I = optional extension

문제:
- 지금 D/E/F는 아직 실제 내용보다 "왜 필요한가" 설명이 더 크다.
- Table ?? / placeholder가 많아 방어가 아니라 약점 노출이 된다.
- Figure numbering/placement도 appendix에서 시각적으로 약간 산만하다.

추천 정리:
1) A/B/C/E/G/H는 살린다.
2) D/F/I는 실제 결과가 약하면 과감히 줄이거나 online appendix로 미룬다.
3) placeholder section은 절대 남기지 않는다.
4) appendix의 각 섹션 첫 문단은 3~4문장 이내로 줄인다.
   지금은 "the intended appendix message"류가 많다.

============================================================
8. format / layout / presentation 지적
============================================================

1) main table은 아직 Improve 중심이다.
- 미팅 때의 방향과도 조금 어긋난다.
- Avg. Rank가 더 설득력 있다.
- 적어도 Table 1과 4.2 본문은 서로 일치해야 한다.

2) placeholder table/caption은 절대 금지
- Table ??, placeholder summary, "-" 표
- 제출 직전 가장 위험한 요소다.

3) appendix figure 배치가 조금 산만하다
- Figure 7/8/9/10 순서가 텍스트와 시각 배치상 아주 매끈하진 않다.
- 가능하면 appendix에서도
  C → D/E → G → H → I
  순서대로 figure/table이 등장하게 정리하는 게 좋다.

4) page count가 많이 늘었다
- 18 pages인데 venue page limit 확인 필요.
- appendix 포함 총량이 괜찮은지 체크 필요.
- 만약 제한이 빡세면, 본문 meta-commentary 줄이는 것만으로도 상당히 줄일 수 있다.

============================================================
10. 최종 판단
============================================================

지금 버전은,
- 예전보다 훨씬 설득력 있고
- challenge → method → experiment의 큰 흐름도 살아 있으며
- reviewer가 "도대체 무슨 논문인지 모르겠다"라고 하긴 어렵다.

하지만 동시에,
- experiment/appendix 통합 과정의 흔적이 아직 많이 남아 있고
- 특히 본문은 "논문"보다 "논문을 어떻게 방어할지에 대한 설명문"처럼 읽히는 부분이 있다.

즉 현재 상태는:
"좋은 논문에 거의 다 왔는데, 아직 final prose가 아니라 merge draft 같다."

핵심만 다시 말하면,
- 지금 가장 큰 문제는 아이디어 부족이 아니다.
- 가장 큰 문제는 "완성된 결과 보고 문장"과 "내부 설계/방어 메모"가 아직 덜 분리된 것이다.

그래서 다음 수정의 목표는
"더 새로운 내용을 추가하는 것"이 아니라,
"이미 있는 좋은 논리와 실험을 최종 제출용 문장으로 정제하는 것"
이어야 한다.
