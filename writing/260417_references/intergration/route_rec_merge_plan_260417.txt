The current manuscript is already much closer to the meeting direction than the earlier versions:

- Method is now aligned with the introduction: Portable Behavioral Cues -> Coarse-to-Fine Multi-Stage Routing -> Behavior-Guided Expert Allocation.

That said, the remaining integration work is not “add more content everywhere.”
It is mostly about tightening role separation and making the paper read like one coherent argument:
- Introduction defines the 3 challenges.
- Method proposes 3 aligned design axes.
- Experiments answer whether each axis matters.
- Appendix supports transparency, reproducibility, and reviewer objections without repeating the main text.

2. Main integration principle
Use the current PDF as the base manuscript.
Then integrate experiment/appendix edits under the following rule:
- Keep anything that strengthens transparency, protocol justification, or challenge-to-evidence mapping.
- Remove anything that repeats the same claim in a weaker or more editorial way.
- Push detailed but non-load-bearing implementation specifics to appendix tables.
- Keep the main text focused on the “why this design / why this evidence” story.

3. How to align with the original meeting points
The meeting had a few recurring priorities. The merged version should explicitly satisfy them:

3.1 Three big ideas must remain visible throughout
Keep the paper organized around:
1) Portable Behavioral Cues
2) Coarse-to-Fine Multi-Stage Routing
3) Behavior-Guided Expert Allocation

Do not allow the experiment/app appendix material to drift into a different framing such as:
- “we tried many variants and this one works”
- “the architecture sweep found this”
without reconnecting it to the 3-axis story.

Every major experiment subsection should answer one of the axes:
- 4.2 Overall performance: Does the whole design help in a fair, transparent setup?
- 4.3 Routing control: Why behavior-guided routing instead of hidden-only or mixed?
- 4.4 Staged structure: Why multi-stage scope separation instead of generic FFN replacement?
- 4.5 Cue sufficiency: Are lightweight portable cues enough?

This is already mostly present; preserve it.

3.2 Transparency of setting must stay explicit
Keep these visible in the merged version:
- subset disclosure in the main table and appendix dataset table
- sessionization thresholds by dataset
- held-out splitting / cleanup explanation
- same validation-selection rule for baselines and RouteRec

Do not move subset disclosure only to appendix. It belongs in the main table and experimental setup.

3.3 Session split / protocol justification should be framed as deliberate, not accidental
The meeting concern was not just “state the setting,” but “make it look justified and defensible.”
So keep the current protocol explanation in 4.1 / Appendix B, but strengthen it with the logic:
- sessions are the unit of evaluation because the paper studies session-aware behavioral routing
- the split keeps temporal order at the session level
- cleanup preserves held-out realism while avoiding uncontrolled leakage from unseen non-target rows

This should read as a methodological choice, not a workaround.

3.4 Feature contribution must stay central
The meeting emphasized that feature design itself is a major contribution.
So when integrating appendix and experiment edits, do not let Portable Behavioral Cues become “just one method subsection.”
Give it support in three places:
- main method section (concept and design rationale)
- cue-family ablation in main experiments
- representative cue bank table in appendix

This triad should stay intact.

4. What to keep in main text vs appendix
This is the biggest integration issue.
There is already some overlap between method, experiments, and appendix. The merge should reduce duplication by function.

4.1 Main text should keep only what is needed to carry the argument
Method main text should keep:
- the common stage interface
- stage windows / cue-family idea
- serial layout logic
- router composition logic
- objective logic

Method main text should NOT become a configuration dump.
Table 2 / Table 3 / Table 4 should carry the more implementation-oriented details.

4.2 Appendix A should be reproducibility- and rationale-oriented
Appendix A should contain:
- fixed main configuration summary
- implementation specification table
- representative cue bank

But it should not repeat the same conceptual sentences already said in 3.1–3.4 unless the appendix version adds reproducibility value.

Concretely:
- A.1 Main Paper Configuration: keep, because it explains why the two-attention placement and macro-mid adjacency are the selected design.
- A.2 Implementation Specification: currently empty / placeholder-like. This must either be populated or removed.
- A.3 Representative Behavioral Cue Families: keep, because it supports the feature contribution.

Recommendation:
Fold Table 3 directly into A.2 and add 2–4 sentences explaining how to read it. That will make Appendix A feel intentional rather than partially unfinished.

4.3 Appendix B should remain the protocol appendix
Appendix B is the right place for:
- final processed dataset statistics
- why subset statistics differ from raw dataset sizes
- evaluation split mechanics
- full cutoff grid

Do not re-explain all of this in the main text. The main text only needs the short defensible protocol summary; Appendix B carries the exact operational details.

4.4 Appendix C / D / E / F / G should each answer one reviewer need
Keep the appendix sections, but make their roles sharper:
- C = architecture and stage/layout ablations
- D = objective and regularization variants
- E = routing diagnostics
- F = behavior-slice analysis
- G = transfer results

Right now these sections are directionally good, but some parts still read like “what the appendix is intended to show.”
Replace editorial intention sentences with concrete observation-oriented framing when final results are ready.

6. Main text–appendix linking: how to connect method and experiment better
This is one of the most important remaining improvements.
The current paper already has the right ingredients, but the links can be made more explicit.

6.1 Connect 3.2 Portable Behavioral Cues -> 4.5 cue-family ablations -> Table 4
Suggested pattern:
- Method says the cue bank is compact, interpretable, and portable.
- Experiment 4.5 tests whether those cues are actually sufficient.
- Appendix Table 4 gives representative examples so the reader knows what the families concretely mean.

Add a short forward reference at the end of 3.2, e.g.:
“The contribution of these cue families is tested directly in Sec. 4.5, while representative stage-wise summaries are listed in Table 4.”

6.2 Connect 3.3 Multi-Stage Routing -> 4.4 stage/layout ablations -> Appendix C
Suggested pattern:
- Method explains why macro/mid/micro differ by evidence scope and granularity.
- Main experiment tests whether these stages matter and whether dense replacement alone is enough.
- Appendix C carries the broader sweep / additional order variants.

Add a short forward reference near the end of 3.3, e.g.:
“The role of stage placement, removal, and ordering is evaluated in Sec. 4.4 and expanded in Appendix C.”

6.3 Connect 3.4 / 3.5 -> 4.3 routing control + Appendix D/E
Suggested pattern:
- Method defines behavior-guided allocation and route consistency + z-loss.
- Main experiment 4.3 tests whether behavior-based routing is better than hidden-only or mixed routing.
- Appendix D shows whether the selected objective terms actually help.
- Appendix E shows routing diagnostics to support the inspectability claim.

Add two references:
- near the end of 3.4: “Alternative routing controls are compared in Sec. 4.3.”
- near the end of 3.5: “Objective variants and routing diagnostics are reported in Appendices D and E.”

This will make the method feel empirically anchored.

7. What to do about overlapping appendix content
There are a few places where appendix sections partially overlap conceptually. Here is the clean split.

A vs C
- A = selected design and reproducibility summary
- C = ablation evidence about why that design was selected
Do not restate ablation conclusions in A beyond one sentence.

B vs main 4.1
- Main 4.1 = compact protocol summary
- B = exact dataset statistics + extended protocol explanation + full k results
Avoid copying long preprocessing paragraphs into both places.

D vs E
- D = objective terms and whether each regularizer helps
- E = what the learned routes look like
D is causal / variant-focused; E is descriptive / diagnostic-focused.
Keep them separate.

F vs 4.2
- 4.2 = overall aggregate performance and high-level interpretation
- F = where gains localize in behavioral slices
So 4.2 should mention slices only lightly, and leave the detailed localization to F.

G vs conclusion/future work
- G is evidence for transfer potential
- conclusion should mention transfer as future extension only if the appendix evidence is genuinely stable enough
If transfer remains mixed or exploratory, keep it appendix-level and avoid making it sound like a central contribution.
