
RouteRec / FMoE 데이터셋 세팅 방어용 정리 노트
================================================
작성 목적:
- 현재 논문의 dataset/session protocol이 “일반적인 raw sequential recommendation 설정”과 다르다는 점을 prior art에 최대한 잘 연결하기 위한 메모
- exact precedent가 없을 때 무엇을 claim하고 무엇을 claim하지 말아야 하는지 정리
- main paper experiment setting / result discussion / appendix dataset section에 각각 어떤 식으로 쓰면 좋은지 초안 가이드 제공
- 다른 모델/에이전트가 여기서 이어서 더 조사하기 쉽게 관련 논문과 링크를 많이 모아두기

현재 RouteRec 초안의 핵심 프로토콜(요약)
---------------------------------------
현재 초안 기준으로는 대략 아래 흐름이다.

1) interactions를 시간순으로 정렬
2) dataset별 inactivity threshold로 sessionization
   - dense implicit 계열: 짧은 threshold
   - sparse explicit/review 계열: 긴 threshold
3) iterative filtering
   - session length < 5 제거
   - item support < 3 제거
   - 제거 후 다시 session length 재점검
4) session start time 기준 temporal split
   - train/valid/test = 70/15/15
5) valid/test에서 training vocabulary에 없는 item은 별도 처리
   - main metric은 seen target 위주로 계산 가능
   - unseen target 결과는 보조 분석/appendix로 분리 가능

핵심 결론
---------
가장 중요한 결론은 아래다.

[결론 1]
이 전체 프로토콜과 “정확히 동일한” 유명 선행논문을 찾기는 어렵다.
따라서 논문 본문에서는
“we follow the exact protocol of X”
처럼 쓰기보다,
“our protocol combines several established practices from session-based/session-aware/sequential recommendation”
처럼 component-wise 정당화를 하는 편이 훨씬 안전하다.

[결론 2]
방어 포인트는 4개로 나누는 것이 가장 좋다.

(A) sessionization 자체는 널리 쓰인다
(B) support/length filtering도 널리 쓰인다
(C) chronological split은 오히려 leakage 방지 측면에서 더 정당하다
(D) seen-only main evaluation 역시 ID-based next-item recommendation에서 매우 흔하다

[결론 3]
리뷰어에게 가장 위험한 문장은
“we use a standard sequential recommendation protocol”
이다.
이건 피하는 게 좋다.
대신
“we evaluate in a sessionized sequential setting”
또는
“we adopt a session-aware temporal evaluation protocol”
처럼 써야 한다.

[결론 4]
네 setting은 “raw user history 전체를 하나의 uninterrupted sequence로 다루는 canonical SR benchmark protocol”은 아니다.
대신
“cross-session information이 존재하는 sessionized sequential setting”
으로 정확히 위치를 잡는 것이 좋다.

권장 스토리라인
----------------
리뷰어가 제일 먼저 느낄 수 있는 의문은 이거다.

“왜 원래 user sequence를 임의로 잘라서 독립 session으로 보고 쓰는가?
그럼 baseline들한테 불리한 거 아닌가?
이 setting이 자의적이지 않나?”

이 질문에 대한 스토리라인은 아래 순서가 제일 좋다.

1. 우리는 raw interaction log를 ‘현실적인 usage sessions’ 관점에서 재구성한다.
   - 특히 RouteRec은 user/session/time-scale behavior를 다루므로,
     uninterrupted full history보다 sessionized history가 모델링 목적과 더 직접적으로 맞닿아 있다.

2. session boundaries는 oracle이 아니라 inactivity-based heuristic으로 정한다.
   - 이는 session-based / session-aware recommendation literature에서 흔한 방식이다.
   - threshold는 one-size-fits-all 상수가 아니라 domain/logging regime에 따라 달라질 수 있다.

3. filtering은 arbitrary pruning이 아니라,
   - 너무 짧아 next-item prediction signal이 약한 sessions 제거,
   - 너무 희귀해 학습/평가가 불안정한 items 제거,
   - cross-session signal이 거의 없는 users 제거
   를 위한 전형적인 안정화 단계다.

4. split은 random이 아니라 chronology-preserving temporal split이다.
   - 즉 “better benchmark score”를 위한 setting이 아니라 leakage를 줄이기 위한 setting이라고 말할 수 있다.

5. unseen target을 main metric에서 제외하는 것은
   - ID-based recommenders가 training vocabulary 밖의 item을 rank할 수 없기 때문이며,
   - prior work에서도 흔히 하는 처리다.
   - 다만 appendix에서 unseen-target statistics나 separate result를 추가로 보여주면 더 좋다.

논문에서 claim해야 할 것 / 피해야 할 것
--------------------------------------
[강하게 claim 가능]
- sessionization by inactivity heuristic is well-established in session-based recommendation
- temporal ordering should be respected to reduce leakage
- filtering short sessions / low-support items is standard practice in session-based benchmarks
- filtering unseen items/targets from held-out evaluation is common for ID-based recommenders

[조심해서 claim]
- dataset-specific threshold choice
- iterative filtering until convergence
- session count threshold (e.g., <=5 sessions drop)
- 70/15/15 exact ratio

이런 것들은 “widely used exact standard”라기보다
“design choice motivated by prior practice + fairness/stability concerns”
로 쓰는 편이 좋다.

[피해야 할 claim]
- “This is the standard sequential recommendation setting.”
- “This protocol exactly follows prior work.”
- “30 minutes / 14 days are canonical thresholds.”
- “Our split ratio is standard.”

추천 reference map (핵심 / 보조)
================================

A. Sessionization / inactivity threshold
----------------------------------------
A1. Jannach, Quadrana, Cremonesi. Session-Based Recommender Systems. Recommender Systems Handbook 2022.
중요도: 매우 높음 (가장 기본 reference)
무엇을 받쳐주나:
- session-based recommendation에서 session이 명시적으로 있지 않으면 post-hoc heuristic으로 만들 수 있다는 점
- typical heuristic으로 “extended inactivity”, 예: 30 minutes를 드는 점
어떻게 쓰면 좋나:
- “we follow common SBRS practice and construct sessions post hoc using inactivity-based heuristics”
링크:
- Handbook PDF: https://web-ainf.aau.at/pub/jannach/files/BookChapter_RS_Handbook_Sessions.pdf
- DOI: https://doi.org/10.1007/978-1-0716-2197-4_8
핵심 메모:
- 이 reference는 exact preprocessing recipe보다 ‘세션을 inactivity로 나누는 것 자체가 정상적’이라는 가장 강한 일반 논거다.

A2. Using autoencoders for session-based job recommendations. UMAP / UMUAI 2020.
중요도: 보조지만 꽤 유용
무엇을 받쳐주나:
- “30-minute idle threshold (as in Quadrana et al. 2017)”라고 직접 연결
- 후속 논문들이 Quadrana류 personalized session-based setting을 30-minute sessionization과 함께 재사용하고 있다는 증거
링크:
- Springer page: https://link.springer.com/article/10.1007/s11257-020-09269-1
- ResearchGate PDF page: https://www.researchgate.net/publication/342609348_Using_autoencoders_for_session-based_job_recommendations
핵심 메모:
- 직접적인 original source는 아니지만, “Quadrana 2017 계열 = 30-min idle threshold”를 간접적으로 확인해주는 좋은 bridge source.

A3. Learning Recommendations from User Actions in the Item-poor Insurance Domain. RecSys 2022.
중요도: 보조, 하지만 narrative용으로 좋음
무엇을 받쳐주나:
- sparse domain에서는 e-commerce/media처럼 짧은 session threshold가 항상 맞는 것이 아니라는 논리
- sparse, long-gap domains에서 session definition이 domain-specific일 수 있다는 논리
링크:
- DOI landing: https://doi.org/10.1145/3523227.3546775
- KU research portal: https://researchprofiles.ku.dk/en/publications/learning-recommendations-from-user-actions-in-the-item-poor-insur
핵심 메모:
- 만약 beauty/review 계열에 긴 threshold를 쓴 이유를 설명해야 하면, 이런 sparse-domain paper를 “threshold is data-regime dependent” narrative에 붙이면 좋다.

A4. Q. Liu & E. Agichtein. Burstiness of Individual Answering Activities / session timeout threshold discussion (ECIR 2011 proceedings snippet)
중요도: 약한 보조
무엇을 받쳐주나:
- 30~40분 수준 session timeout heuristic이 사용자 activity burst 분석에서 나오는 오래된 근거
링크(검색 snippet 기반):
- https://dokumen.pub/advances-in-information-retrieval-33rd-european-conference-on-ir-resarch-ecir-2011-dublin-ireland-april-18-21-2011-proceedings.html
핵심 메모:
- recommender paper는 아니므로 핵심 reference로 쓰기보다는 “30min heuristic의 고전적 감각” 정도로만 참고.

논문 본문에서 experiment setting을 어떻게 쓰면 좋은가
=================================================

1) Main paper - Setting subsection에서의 기본 전략

핵심은 “이건 standard seqrec setting”이라고 우기지 말고,

권장 서술 포인트:
- RouteRec은 user/session/time-scale behavioral control을 다루므로,
  raw uninterrupted history보다 sessionized histories가 modeling assumption과 더 잘 맞음
- explicit session IDs가 없는 corpora에서는 inactivity-based sessionization을 사용
- 30-min heuristic is common for dense logs, but sparse datasets may require longer thresholds
- short sessions and low-support items are iteratively removed to ensure stable next-item supervision
- we use session-level chronological split to reduce temporal leakage
- main tables report seen-target performance; unseen-target cases are analyzed separately / in appendix

메인 setting paragraph용 영어 초안(짧은 버전)
--------------------------------------------
Across datasets, we first sort interactions chronologically and construct sessions using inactivity-based heuristics, following common practice in session-based and session-aware recommendation. We use shorter thresholds for dense implicit-feedback logs and longer thresholds for sparse review-like logs, based on inter-event statistics and domain characteristics rather than assuming a universal cutoff. After sessionization, we iteratively remove sessions that are too short for reliable next-item supervision and items with extremely low support, so that the final corpus satisfies the minimum session-length and item-support constraints consistently. We then split sessions chronologically at the session level into train/validation/test partitions, which better respects temporal order and reduces leakage than non-temporal protocols. Because ID-based next-item recommenders cannot rank items absent from the training vocabulary, our main reported metrics are computed on held-out cases with seen targets; statistics and supplementary analyses for unseen-target cases are provided separately.

메인 setting paragraph용 영어 초안(조금 더 방어적인 버전)
------------------------------------------------------
Our goal is not to reproduce the exact raw-data protocol of prior sequential recommendation benchmarks, but to evaluate all methods under a unified sessionized sequential setting that exposes both within-session signals and cross-session context. This choice is aligned with prior session-based and session-aware work, where usage sessions are commonly constructed from timestamped logs using inactivity heuristics when oracle session IDs are unavailable. We therefore adopt dataset-dependent inactivity thresholds, apply iterative support and length filtering for stability, and use a session-level chronological split to preserve temporal order. Since all compared methods are trained and evaluated under the same processed data, the comparison isolates the effect of the model rather than differences in preprocessing.

2) Main paper - results analysis에서의 전략
----------------------------------------
results analysis에서는 “우리가 기존 논문보다 더 높은 수치를 찍었다”보다,
“우리 모델이 이 sessionized setting에서 behavior-guided routing을 통해 일관되게 이득을 보였다”를 강조하는 것이 좋다.

결과 분석에서 넣으면 좋은 문장 포인트:
- The gains are measured under a unified sessionized evaluation protocol shared across all baselines.
- Therefore, improvements should be interpreted as evidence that behavior-guided routing is effective in this setting, not as a claim about raw-protocol leaderboard dominance.
- Because preprocessing choices can materially affect sequential evaluation, we explicitly document the protocol and supplementary statistics in the appendix.
- If possible, add one sentence:
  “We do not treat the preprocessing pipeline as incidental; rather, it defines the behavioral granularity at which the task is evaluated.”

results section에 넣기 좋은 짧은 caution 문장
-------------------------------------------
All results are reported under the same sessionized temporal protocol for every method. We therefore interpret the comparison as model effectiveness within this evaluation setting, rather than as a direct replacement for raw-sequence benchmarks using different preprocessing choices.

혹시 reviewer 방어용으로 더 강하게 쓰고 싶다면
-------------------------------------------
Recent work has shown that sequence/session splitting choices can substantially affect offline evaluation outcomes. We therefore make the protocol explicit and keep it fixed across all methods, so that the experimental comparison reflects modeling differences rather than hidden preprocessing variance.

Appendix용 영어 bullet 초안
--------------------------
For transparency, we report the complete preprocessing pipeline for each dataset, including (i) the inactivity threshold used for session construction, (ii) the number of interactions, sessions, users, and items before and after filtering, (iii) summary statistics of session length and sessions per user, and (iv) the proportion of held-out cases with unseen targets/items. This documentation is important because recent work has shown that splitting and preprocessing choices can materially affect offline sequential recommendation results.

Appendix dataset paragraph용 영어 초안
------------------------------------
The preprocessing pipeline is unified in structure but dataset-specific in temporal granularity. We first sort each user’s interactions by timestamp and construct sessions using an inactivity threshold chosen according to the logging regime of the dataset. Dense implicit-feedback corpora are segmented with shorter thresholds, whereas sparse review-like corpora use longer thresholds. After sessionization, we iteratively remove sessions shorter than the minimum length threshold and items with support below the minimum count threshold until convergence. We then sort sessions by start time and split them chronologically into 70%/15%/15% train/validation/test partitions. Because ID-based recommenders cannot score items outside the training vocabulary, our main metrics are computed on held-out examples with seen targets, while unseen-target statistics are reported separately.

추천 robustness / 추가 실험 (가능하면)
======================================
이건 꼭 해야 하는 건 아니지만, reviewer 방어력이 급상승한다.

R1. threshold sensitivity
- 1~2개 dataset에서 inactivity threshold를 조금 바꿔서
  RouteRec vs baseline ranking이 크게 뒤집히지 않는지 확인
- 예: 20/30/40 min 또는 7/14/21 day

R2. split sensitivity
- 1~2개 dataset에서 80/10/10 혹은 70/10/20 등으로 살짝 바꿔보기
- 혹은 per-user-last-session split 비교

R3. seen vs all / unseen
- main metric은 seen-only로 두되
- appendix에 unseen-target 포함 결과 또는 unseen-only slice를 추가
- “we are not hiding it; we separate it because the task differs”

R4. session count filter sensitivity
- session<=3 drop vs <=5 drop 비교
- especially user-history-aware model에서 cross-session info가 어느 정도 있어야 의미가 생기는지 보여줄 수 있음

R5. baseline fairness note
- full-history sequential baselines에 대해서,
  본 논문에서는 모든 모델을 동일한 processed sessions 위에서 학습/평가한다고 명시
- 가능하면 baseline input construction을 appendix에 자세히 적기
  (e.g., each session prefix predicts next item within that session; no model receives extra raw-history access not available to others)

마지막 실무 조언
==============
- 네 protocol은 “이상한 임의 설정”으로 보일 수도 있지만,
  사실 component-wise로 보면 선행연구의 관행들을 꽤 많이 공유한다.
- 문제는 “전체를 standard라고 부르면” 공격당한다는 점이다.
- 따라서 positioning만 바르게 하면 된다.

제일 안전한 한 문장:
We evaluate all methods under a unified sessionized temporal protocol, motivated by established practices in session-based and session-aware recommendation, and make the full preprocessing pipeline explicit because recent work has shown that split/session construction choices can materially affect offline results.

한글로 요약하면:
“우리는 이 프로토콜이 standard raw-sequence SR benchmark라고 주장하는 것이 아니라, session-based/session-aware literature에 기반한 unified sessionized temporal protocol이라고 명시하고, 그 선택이 결과에 영향을 줄 수 있음을 인정한 상태에서 전처리 전체를 투명하게 공개한다.”

이 framing이 현재로서는 가장 방어력이 높다.
