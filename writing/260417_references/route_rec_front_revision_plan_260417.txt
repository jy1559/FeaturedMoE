RouteRec 논문 미팅 결과 및 Abstract~Related Work 수정 계획 정리
작성일: 2026-04-17

[문서 목적]
이 문서는 최근 미팅 내용과 현재 논문 초안 상태를 바탕으로,
1) 미팅 결과를 깔끔하게 재정리하고,
2) abstract, introduction, related work를 어떤 방향으로 수정하면 좋을지,
3) 이후 method와 experiments까지 자연스럽게 이어지도록 앞부분의 논리 구조를 어떻게 다시 잡을지
정리한 메모이다.

--------------------------------------------------
1. 전체 판단: 지금 논문이 부족한 것은 아이디어가 아니라 설득 구조
--------------------------------------------------

현재 초안의 핵심 아이디어 자체는 이미 존재한다.
- behavior cue를 hidden representation과 분리하여 routing에 사용한다.
- macro / mid / micro의 세 temporal scope를 둔다.
- compact behavior feature를 이용해 expert routing을 behavior-grounded하게 만든다.

문제는 이 아이디어들이 "왜 필요한가", "왜 이렇게 설계했는가", "왜 이 세 축이 서로 연결되는가"가 앞부분에서 충분히 설득되지 않는다는 점이다.
즉, 설계는 있는데 설계 이유가 요약본처럼 짧게 지나가며, 현재 front-half는 좋은 아이디어 소개문에 가깝고 논문식 problem-solution structure는 아직 약하다.

따라서 이번 수정의 핵심 방향은 다음과 같다.
- 논문을 3개의 challenge와 3개의 solution axis로 재정렬한다.
- abstract, introduction, related work, method overview가 같은 축을 공유하게 만든다.
- preliminary는 제거하고, 대신 introduction과 method 초반의 설명 밀도를 높인다.
- 이후 ablation과 analysis도 같은 3축을 검증하는 구조로 맞춘다.

--------------------------------------------------
2. 미팅 결과 핵심 정리
--------------------------------------------------

2.1 서술 및 구조 관련
- methodology까지의 앞부분이 현재 너무 summary 중심이다.
- "왜 이런 feature를 만들었는지", "왜 stage를 나눴는지", "왜 routing을 이렇게 했는지"가 더 자세히 보여야 한다.
- preliminary는 빼는 편이 낫다.
- problem statement는 method 앞부분 혹은 method 초반으로 흡수한다.
- MoE background는 related work 쪽에 짧게 녹이는 것이 낫다.
- method의 전개 순서는 stage 먼저, routing 나중이 더 자연스럽다.
- subsection 이름도 contribution 구조가 바로 보이게 정리해야 한다.
- overview에서부터 problem/challenge를 먼저 던지고, 각 challenge에 대한 답으로 방법을 설명하는 구조가 좋다.

2.2 논문의 큰 설계축 관련
- 이 논문은 하나의 단일 novelty보다, 서로 맞물린 3개의 설계 아이디어로 쓰는 편이 낫다.
- 이후 method subsection, ablation, qualitative analysis도 이 3축을 따라 정리하는 것이 통일성이 좋다.
- introduction에서도 이 3축을 초반에 예고하고, method에서 다시 구체화하는 방식이 좋다.

2.3 실험/보고 방식 관련
- sampled subset 사용 사실은 main table에서 바로 보이게 해야 한다.
- session split / sessionization은 지금 와서 크게 바꾸기보다, 왜 그렇게 했는지 근거를 appendix로 보강하는 편이 낫다.
- hparam search space와 selection rule을 솔직하게 적어 reviewer 질문에 대비해야 한다.
- dense routing이면 정말 MoE라고 부를 수 있는지 다시 확인할 필요가 있다.
- main table에서 improvement %만 강조하기보다 avg ranking, 통계 검정 등 더 납득되는 reporting도 고려할 수 있다.
- table caption 위치, dataset description 방식, sampled/full 구분 등 기본적인 presentation도 더 깔끔해야 한다.

2.4 추가 실험/분석 관련
가장 중요한 우선순위:
- feature ablation
- hidden-only routing과의 비교
- feature engineering에 대한 ablation

그 다음 우선순위:
- stage ablation
- routing design ablation
- case study 혹은 직관적인 qualitative evidence

핵심은 "성능이 오른다"를 넘어서,
- feature가 routing에 실제로 의미 있게 쓰였는가?
- stage 분리가 실제로 유효한가?
- routing이 의도한 behavior regime을 어느 정도 반영하는가?
를 보여주는 것이다.

--------------------------------------------------

3.1 Abstract의 문제
- 문제의식과 방향은 좋지만 메시지가 약간 분산된다.
- readability / preliminary formulation 같은 포인트는 contribution으로 약하다.
- abstract는 problem -> gap -> method -> evidence 구조로 더 날카롭게 가야 한다.

3.2 Introduction의 문제
- 전체 방향은 맞다.
- 하지만 현재는 challenge-solution 대응이 명시적으로 정리되어 있지 않다.
- 각 문단이 좋은 이야기처럼 이어지지만, 독자가 "그래서 이 논문이 정확히 어떤 3개의 문제를 풀고 무엇으로 답하는가"를 빠르게 잡기 어렵다.

3.3 Related Work의 문제
- 단순 분류는 되어 있으나, 우리 문제의식과 대응되는 식으로 재정렬되지는 않았다.
- 앞으로는 단순히 분야를 나누는 것이 아니라,
  "기존 연구는 challenge 1, 2, 3을 각각 어디까지 다루었고 어디가 비어 있는가"를 보여주는 구조가 더 좋다.

--------------------------------------------------
4. 논문 전체를 관통할 challenge 3개
--------------------------------------------------

Challenge 1. Behavioral diversity exists, but computation is still mostly shared.
설명:
사용자와 세션은 실제로 서로 다른 behavioral regime을 보인다. 어떤 경우는 stable preference가 강하고, 어떤 경우는 session intent가 중요하며, 어떤 경우는 immediate transition이나 local exploration이 중요하다. 그러나 기존 sequential recommender는 대부분 shared computation path 위에서 이를 처리한다.
핵심 문제:
heterogeneous behavior와 shared computation 사이의 mismatch.

Challenge 2. Behavioral evidence exists at different temporal scopes, but routing is usually not scope-aware.
설명:
행동 단서는 long-term preference, within-session evolution, immediate local transition처럼 서로 다른 시간 범위에서 나타난다. 그런데 하나의 feature view 혹은 하나의 routing bias로 모든 horizon을 처리하면 coarse signal과 fine signal을 동시에 잘 반영하기 어렵다.
핵심 문제:
temporal scope mismatch.

Challenge 3. Hidden-state-based routing can work, but its semantics are opaque and hard to inspect.
설명:
MoE를 사용하더라도 routing을 hidden state에만 의존하면 expert allocation이 어떤 observable behavior와 대응되는지 해석하기 어렵다. 성능은 나더라도 controllability와 diagnosability가 약하다.
핵심 문제:
opaque routing semantics.

--------------------------------------------------
5. method의 중심축 3개 (challenge 3개에 대응)
--------------------------------------------------

아래 3개 축으로 front-half부터 method까지 통일해서 미는 것이 좋다.

Axis 1. Portable Behavioral Cues
핵심 의미:
ordinary interaction logs에서 얻을 수 있는 compact하고 transferable한 behavior cue를 만든다.
포인트:
- rich side information dependence를 줄인다.
- feature engineering의 contribution을 "많이 만들었다"가 아니라 "작고 portable한 cue bank를 설계했다"로 설명한다.
대응 challenge:
Challenge 1, 3의 기반.

Axis 2. Coarse-to-Fine Multi-Stage Routing
핵심 의미:
macro / mid / micro의 세 scope를 두고, 서로 다른 temporal evidence를 서로 다른 granularity로 반영한다.
포인트:
- long-term, session-level, immediate local signal을 구분한다.
- stage 분리의 이유를 단순 architecture choice가 아니라 temporal scope alignment로 설명한다.
대응 challenge:
Challenge 2의 직접적인 해법.

Axis 3. Behavior-Guided Expert Allocation
핵심 의미:
hidden representation과 routing signal을 분리하고, behavior cue가 expert path를 제어하게 만든다.
포인트:
- hidden state는 content transformation에 사용한다.
- behavior cue는 routing control에 사용한다.
- 해석 가능성, diagnosability, behavior-grounded specialization을 강조한다.
대응 challenge:
Challenge 3의 직접적인 해법.

요약 대응:
- Challenge 1 -> Portable Behavioral Cues
- Challenge 2 -> Coarse-to-Fine Multi-Stage Routing
- Challenge 3 -> Behavior-Guided Expert Allocation

주의:
정확히는 축 1이 challenge 1을 완전히 단독 해결한다기보다, 축 1이 축 2와 3의 기반 cue interface를 제공하는 역할이다. 그러나 front-half에서는 위 대응으로 정리해도 충분히 자연스럽다.

6. Abstract 수정 계획
--------------------------------------------------

6.1 Abstract에서 반드시 할 일
- preliminary / readability 관련 언급 제거
- problem -> gap -> method -> evidence 구조를 선명하게 만들기
- 3개의 design axis를 한 문단 안에서 자연스럽게 요약하기
- "ordinary logs로도 충분히 routing signal을 만들 수 있다"는 점을 강조하기

6.2 Abstract의 권장 논리 흐름
1) Sequential recommendation is behaviorally heterogeneous, but computation is mostly shared.
2) MoE is promising, but hidden-only routing is opaque and not naturally scope-aware.
3) We propose RouteRec, built on three ideas:
   - Portable Behavioral Cues
   - Coarse-to-Fine Multi-Stage Routing
   - Behavior-Guided Expert Allocation
4) RouteRec uses compact cues from ordinary logs and decouples routing from representation.
5) Experiments show competitive/improved ranking quality and more behavior-consistent routing patterns.

6.3 Abstract에서 강조할 핵심 메시지
- 이 논문은 rich metadata modeling paper가 아니다.
- 이 논문은 behavior cue를 routing control signal로 쓰는 conditional computation paper이다.
- 세 stage와 세 cue family/temporal scope는 임의의 장식이 아니라 challenge에 대응하는 구조적 답이다.

7. Introduction 수정 계획
--------------------------------------------------

권장 구조: 5문단

Paragraph 1. Hook: user/session behavior is not homogeneous.
목표:
- next-item prediction처럼 보이지만 실제 behavior는 heterogeneous하다는 점을 직관적으로 제시한다.
- stable preference / session intent / local exploration 또는 local transition을 예시로 든다.
- Figure 1과 자연스럽게 연결한다.
연결 challenge:
Challenge 1.

Paragraph 2. Existing sequential recommenders are strong, but mostly rely on shared computation.
목표:
- 기존 backbone들이 sequence modeling 성능은 강하다는 점을 인정한다.
- 하지만 behaviorally distinct cases에 computation을 다르게 배분하는 explicit mechanism은 약하다고 정리한다.
- 여기서 shared computation의 한계를 분명히 만든다.
연결 challenge:
Challenge 1.

Paragraph 3. Existing heterogeneous-state / feature-aware approaches only partially address the issue.
목표:
- session intent, multi-interest, feature-aware SR이 heterogeneity와 extra signals를 다루고 있음을 인정한다.
- 하지만 대부분 representation enrichment, fusion, attention modulation 수준에 머물고,
  feature를 computation path control signal로 쓰는 방향은 약하다고 대비한다.
- 여기서 cue vs representation의 역할 분리를 예고한다.
연결 challenge:
Challenge 1, 3.

Paragraph 4. MoE is a natural candidate, but hidden-state routing remains limited.
목표:
- conditional computation / MoE가 가능성 있는 방향임을 인정한다.
- 그러나 hidden-state-based routing은 semantics가 opaque하고, temporal scope를 명시적으로 반영하기 어렵다고 말한다.
- 여기서 challenge 2와 challenge 3을 명시적으로 제시하는 것이 좋다.
예시 문장 흐름:
  This raises three practical challenges: how to derive portable routing cues from ordinary logs, how to align routing with different temporal scopes, and how to make expert allocation behavior-grounded and inspectable.

Paragraph 5. Our answer: RouteRec.
목표:
- RouteRec을 세 축으로 소개한다.
  1) Portable Behavioral Cues
  2) Coarse-to-Fine Multi-Stage Routing
  3) Behavior-Guided Expert Allocation
- 세 축이 각각 무엇을 담당하는지 한두 문장씩 설명한다.
- 마지막에 contribution summary로 자연스럽게 연결한다.

7.1 Introduction 전체 톤 조정 방향
- 기존보다 조금 더 자세하게 간다.
- 단순 motivational writing이 아니라 challenge-solution mapping이 보이게 쓴다.
- method와 실험을 미리 예고하는 역할을 하게 만든다.
- 특히 macro/mid/micro와 feature/routing decoupling이 이후 method에서 다시 등장할 수 있게 terminology를 introduction에서 미리 정해둔다.

8. Related Work 수정 계획 (3개 정렬 구조)
--------------------------------------------------

Related work는 challenge 3개와 통일되도록 3개 subsection으로 재편한다.

8.1 Section 1: Shared-Path Sequential Recommendation
다룰 내용:
- GRU4Rec, SASRec, TiSASRec, BSARec, SIGMA, DuoRec 등 strong sequential recommenders
핵심 정리:
- representation quality를 크게 개선했지만, 대부분 shared computation path를 전제로 한다.
- heterogeneous behavior를 조건별로 다른 computation path에 연결하는 explicit mechanism은 약하다.
우리 논문과의 차이:
- 우리는 strong backbone 위에서 computation allocation 자체를 conditionally 바꾸는 문제를 다룬다.
대응 challenge:
Challenge 1.

8.2 Section 2: Heterogeneous Behavior and Multi-Scope Signal Modeling
다룰 내용:
- session intent, multi-interest, long-short preference, feature/time-aware modeling 관련 연구
- heterogeneity를 인정하고 여러 신호를 이용하려는 접근들
핵심 정리:
- 기존 연구는 user/session state가 단일하지 않다는 사실을 보여주었다.
- 그러나 대부분 representation decomposition, attention reweighting, side-information fusion 중심이다.
- temporal scope를 명시적으로 나누어 routing control 문제로 연결한 경우는 드물다.
우리 논문과의 차이:
- 우리는 heterogeneity를 representation issue가 아니라 routing/control issue까지 확장한다.
- 또한 macro/mid/micro를 coarse-to-fine multi-stage routing으로 구조화한다.
대응 challenge:
Challenge 2 중심, 일부 Challenge 1.

8.3 Section 3: Expert Specialization and Conditional Computation in Recommendation
다룰 내용:
- MoE, conditional computation, recommendation에서의 expert specialization 관련 연구
핵심 정리:
- expert specialization은 유망한 방향이다.
- 그러나 hidden-driven routing이나 task/facet 중심 specialization은 observable behavioral cue와의 연결이 약할 수 있다.
- routing semantics와 inspectability가 제한된다.
우리 논문과의 차이:
- 우리는 routing signal을 hidden representation에서 분리하고,
  portable behavior cue를 통해 expert allocation을 behavior-grounded하게 만든다.
대응 challenge:
Challenge 3.

8.4 Related Work 전체 작성 원칙
- 단순 문헌 분류로 끝내지 않는다.
- 각 subsection 마지막 문단에서 "무엇을 했는지"보다 "우리 challenge를 어디까지 다뤘고, 무엇이 남아 있는지"를 정리한다.
- introduction과 vocabulary를 맞춘다. (shared computation, temporal scope, inspectable routing, portable cues 등)

10. 이 수정이 이후 method와 어떻게 연결되는가
--------------------------------------------------

앞부분을 위처럼 정리해두면 method도 매우 자연스럽게 내려간다.

Method 연결 예시:
- Problem setup / overview
- Portable Behavioral Cues
- Coarse-to-Fine Multi-Stage Routing
- Behavior-Guided Expert Allocation
- Training Objective / Stabilization

즉 front-half에서 소개한 3축이 그대로 method subsection으로 이어질 수 있다.
그리고 ablation도 같은 축으로 대응시키기 쉽다.
- cue ablation -> Axis 1 검증
- stage ablation -> Axis 2 검증
- routing control comparison -> Axis 3 검증

12. 최종 요약
--------------------------------------------------

이번 미팅을 반영한 가장 중요한 방향은 다음과 같다.

1) 이 논문은 하나의 novelty보다 3개의 연결된 설계축으로 정리하는 것이 낫다.
2) front-half는 challenge 3개와 solution axis 3개를 중심으로 다시 써야 한다.

권장 최종 프레이밍:

- Challenge 2: Temporal scope mismatch
- Challenge 3: Opaque hidden-state routing

- Axis 1: Portable Behavioral Cues
- Axis 2: Coarse-to-Fine Multi-Stage Routing
- Axis 3: Behavior-Guided Expert Allocation

이 3x3 대응을 논문 전체의 backbone으로 삼는 것이 현재로서는 가장 좋은 방향이다.
